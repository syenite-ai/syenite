## 2. Known test failures (triage now, don't merge upstream until clean)

### 2.1 `tests/carry.test.ts` — 7 failures

**Root cause:** Track B renamed `getMorphoRates` → `getMorphoRatesMultiChain` in `mcp/src/data/morpho.ts`. The carry-screener mock at `tests/carry.test.ts` line ~180 declares a partial `vi.mock("../src/data/morpho.js")` that only exports the old name.

**Fix:** Extend the mock to also declare `getMorphoRatesMultiChain` (same shape as `getMorphoRates`). Verify `rates.ts` and `carry.ts` both consume the new function. One test file, ~10-minute fix.

**Verify after fix:** `npm test -- tests/carry.test.ts`

### 2.2 `tests/morpho-multichain.test.ts` — 2 failures

Both are of the shape "expected `[]` to deeply equal ...". The test asserts that `getMorphoRatesMultiChain` / `getAllMetaMorphoVaults` return `[]` on GraphQL errors / non-ok responses, but the implementation is returning populated arrays (likely because mocks aren't wired correctly).

**Two possibilities:**
1. The agent's implementation silently returns a synthesized fallback array instead of `[]` on error. **This is a bug per CLAUDE.md ("fail fast")** and must be fixed — error paths should return `[]` and log, not synthesise.
2. The mock isn't actually returning an error response the way the test expects.

**Action:** Read `mcp/src/data/morpho.ts`'s error branches end-to-end. The contract per Track B's brief was "If a Morpho or Pendle API call fails, log and return `[]` — never throw." Verify that's actually what the code does. Fix whichever side is wrong.

### 2.3 `tests/prediction-watch.test.ts` — 1 failure

Test: "resolution-approaching alert fires when market close time is within threshold."

**Root cause (likely):** Track C's `checkPredictionWatch` in `mcp/src/data/alert-checker.ts` either (a) doesn't evaluate the `resolutionApproachingHours` condition on every tick, (b) uses the wrong field for close time, or (c) has a state-dedup bug where the alert fires once and then doesn't re-fire in the test.

**Action:** Read `checkPredictionWatch` (lines ~214 onward in the merged alert-checker.ts). Walk through the test scenario (setup at `tests/prediction-watch.test.ts:218` — expected `resAlert` to be defined after check). Fix the firing logic so resolution-approaching alerts generate regardless of prior firing within the window.

- `lending.rates.query` with `chain: "base"` — does it now correctly include Morpho on Base markets? Previously it only covered Ethereum.

### 4.4 Track C — Prediction loop

**Files to read:**
- `mcp/src/data/polymarket.ts` — new functions `getMarketDetail`, `getMarketPriceHistory` (verify they exist).
- `mcp/src/data/alerts.ts` — `WatchType`, prediction-specific fields on `WatchConfig`.
- `mcp/src/data/alert-checker.ts` — `checkPredictionWatch` logic (lines ~214+).
- `mcp/src/tools/prediction-{market,watch,position,quote,order}.ts` — five new tools.

**Questions:**
- **Ship-blocker:** `prediction.order` returns an `eip712_offchain_order` (per the schema's `mode: z.literal("eip712_offchain_order")`). This means Polymarket execution **is NOT an on-chain tx** — it's an off-chain signed order that gets submitted to their CLOB REST endpoint. This is fundamentally different from the lending execute flow and `tx.verify` / `tx.simulate` / `tx.receipt` will NOT work on it. The build plan anticipated this exact case (the spike at start of C2). Decide:
  1. Ship as EIP-712 signing flow with clear docs and no trust-layer integration (and update `syenite.help` to make this distinction visible to agents), OR
  2. Defer `prediction.order` until we implement a Polymarket-specific trust layer.
- `prediction.watch` stores state in the in-memory watches Map from `mcp/src/data/alerts.ts`. On restart, all prediction watches are lost. Is that acceptable for v0.6 or must we persist? (Lending watches have the same issue today.)
- The resolution-approaching alert firing logic is failing its test (§2.3). Fix before ship.
- Fee estimation in `prediction.quote` — Polymarket charges maker/taker fees. Are the `makerBps` / `takerBps` values hardcoded and current? They've changed before.

Do **not** roll back the `usage_logs` schema migration. Columns are nullable and historic queries still work.
