# Syenite v0.2 — Build Plan

## Goal

Ship the base-tier product that's worth distributing: open access, multi-asset, multi-protocol DeFi lending intelligence. No monetisation yet — measure demand via usage analytics before adding paid tiers.

## Design Decisions

### Auth: optional, not required

**Current:** API key required → agent can't use Syenite without visiting the website, requesting a key, and configuring it. This is a distribution killer.

**Change:** Anonymous access allowed. IP-based rate limiting (20 req/min) for unauthenticated requests. Optional API keys get higher limits (100 req/min). Invalid keys still rejected — no silent degradation.

**Rationale:** Standard in the MCP ecosystem. Most free MCP servers require no auth. Agents should be able to add the URL and start querying immediately. We still get full usage visibility via request logging (by IP or key).

### Multi-asset: query all Aave reserves dynamically

**Current:** Hardcoded BTC wrappers (wBTC, tBTC, cbBTC) as the only collateral options.

**Change:** Query `pool.getReservesList()` on Aave/Spark to discover all active assets dynamically. Let the user filter by collateral type. Support any collateral × any borrow asset pair the protocol supports.

**Coverage after expansion:**
- ETH, WETH, wstETH, rETH, cbETH (ETH ecosystem)
- wBTC, tBTC, cbBTC (BTC ecosystem)
- USDC, USDT, DAI, GHO, BOLD (stablecoins — for supply-side queries)
- Any other Aave-listed asset (sDAI, LINK, AAVE, etc.)

### Multi-protocol: same ProtocolRate interface, different fetchers

Each protocol gets its own data fetcher that returns `ProtocolRate[]`. The tool layer doesn't care which protocol the data comes from — it just aggregates and ranks.

### Infra cost: unchanged

All protocols are on Ethereum mainnet, queried via the same Alchemy RPC. Each new protocol adds ~3-5 RPC calls per request. Alchemy free tier allows 300M compute units/month. Even at 10,000 requests/day across all protocols, we'd use <1% of the free tier. Cost scales with demand only.

## Phases

### Phase 1: Open access + multi-asset (remove friction)

**1a. Make auth optional**
- `src/index.ts`: Allow requests without Authorization header through to MCP
- `src/auth/rate-limit.ts`: Add IP-based rate limiting for anonymous requests (20/min)
- Keep API key validation for requests that include a key
- Log all requests (IP + key if present) for analytics

**1b. Expand asset coverage in Aave v3**
- `src/data/types.ts`: Replace `BtcWrapper` type with generic asset config. Add ETH/LST/stablecoin tokens and decimals.
- `src/data/aave.ts`: Refactor to dynamically query Aave reserves instead of iterating over hardcoded BTC wrappers. For a given borrow asset, find all collateral options.
- Alternative: use Aave's `UIPoolDataProvider.getReservesData()` which returns all reserves in one call (more efficient, fewer RPC round-trips).

**1c. Expand Morpho Blue markets**
- `src/data/types.ts`: Add known Morpho Blue markets beyond BTC (ETH/USDC, wstETH/USDC, wstETH/ETH, etc.)
- `src/data/morpho.ts`: No structural changes needed — just more market configs.
- Note: Morpho markets are identified by hash, so we maintain a curated list. Can use Morpho's public API to discover markets if we want auto-discovery later.

### Phase 2: Add protocols

**2a. Spark (SparkLend)**
- Aave v3 fork — identical ABIs, different contract addresses
- `src/data/spark.ts`: Reuse Aave v3 logic with Spark's Pool, PoolDataProvider addresses
- Spark Pool: `0xC13e21B648A5Ee794902342038FF3aDAB66BE987`
- Spark PoolDataProvider: query from SparkLend PoolAddressesProvider
- Key markets: ETH, wstETH, rETH, sDAI as collateral; DAI/sDAI as borrow
- Effort: Low — copy aave.ts, swap addresses, adjust asset list

**2b. Compound v3 (Comet)**
- Single-asset markets (each Comet is one borrow asset, multiple collaterals)
- `src/data/compound.ts`: New fetcher using Comet ABI
- Key contracts: USDC Comet (`0xc3d688B66703497DAA19211EEdff47f25384cdc3`), WETH Comet (`0xA17581A9E3356d9A858b789D68B4d866e593aE94`)
- Reads: `getAssetInfo()`, `getUtilization()`, `getSupplyRate()`, `getBorrowRate()`, `totalSupply()`, `totalBorrow()`
- Effort: Medium — different ABI but straightforward reads

**2c. Euler v2**
- Vault-based architecture — each vault is a market
- `src/data/euler.ts`: New fetcher using EVault ABI
- Key vaults discoverable via Euler Vault Connector and Euler Lens
- Reads: `interestRate()`, `totalAssets()`, `totalBorrows()`, collateral factor via governance
- Effort: Medium — need to research vault discovery and interest rate model

**2d. Liquity v2**
- Unique architecture: borrow BOLD stablecoin against ETH/LSTs
- `src/data/liquity.ts`: New fetcher using BorrowerOperations and TroveManager ABIs
- Fixed-rate borrowing (user-set interest rate) — different from variable-rate protocols
- Key data: min/max interest rates, total collateral, total BOLD debt, individual trove rates
- Markets: ETH/BOLD, wstETH/BOLD, rETH/BOLD
- Effort: Medium — unique model but well-documented contracts

### Phase 3: Polish for distribution

- Update all tool descriptions to reflect multi-asset, multi-protocol scope
- Update landing page copy and documentation
- Update `.well-known/mcp.json` with expanded tool descriptions
- Update README with new capabilities and examples
- Test end-to-end with Claude Desktop, Cursor, OpenAI Agents SDK
- Submit to MCP directories (Smithery, Glama, mcp.so, PulseMCP, MCP Hub)

### Phase 4: Engagement analytics (post-distribution)

- Log which assets/protocols agents query most
- Simple email signup for optional API key (just a form on landing page)
- Dashboard improvements: query heatmaps, popular assets, error rates
- Consider: webhook/alert when usage crosses thresholds

### Phase 5: Monetisation (future, demand-dependent)

- Execution tools: transaction building, calldata generation
- Premium data: historical rates, yield curve, backtesting
- Position management: automated rebalancing via LISA/MLISA
- Stripe billing for premium tier
- Revenue from interest spreads on managed positions

## Implementation Order

```
1a  Auth optional + IP rate limiting     [~30 min]
1b  Multi-asset Aave v3                  [~2 hr]
1c  Expand Morpho markets               [~1 hr]
2a  Add Spark                            [~1 hr]
2b  Add Compound v3                      [~2 hr]
2c  Add Euler v2                         [~2 hr]
2d  Add Liquity v2                       [~2 hr]
3   Polish + update docs                 [~1 hr]
                                    ─────────────
                              Total: ~12 hr implementation
```

Each phase deploys independently — push to main triggers auto-deploy. We can ship incrementally: auth fix + multi-asset first, then add protocols one at a time.

## Architecture After v0.2

```
src/
├── auth/
│   ├── keys.ts              # Optional API key validation
│   └── rate-limit.ts        # IP + key based rate limiting
├── data/
│   ├── client.ts            # viem RPC client
│   ├── cache.ts             # SQLite cache layer
│   ├── prices.ts            # Chainlink price feeds
│   ├── types.ts             # Shared types, token registry, protocol addresses
│   ├── aave.ts              # Aave v3 fetcher (dynamic reserves)
│   ├── spark.ts             # Spark fetcher (Aave v3 fork)
│   ├── morpho.ts            # Morpho Blue fetcher
│   ├── compound.ts          # Compound v3 fetcher
│   ├── euler.ts             # Euler v2 fetcher
│   └── liquity.ts           # Liquity v2 fetcher
├── tools/
│   ├── rates.ts             # Aggregates all protocol fetchers
│   ├── market.ts            # Cross-protocol market overview
│   ├── monitor.ts           # Position monitoring
│   └── risk.ts              # Risk assessment
├── logging/
├── web/
└── index.ts
```

## What Doesn't Change
- MCP transport (Streamable HTTP, stateless)
- Tool names and response schema (additive only)
- SQLite for cache + logging
- Docker deployment on DO App Platform
- Auto-deploy on push to main
