MCP trust, speed, and security

An MCP server is a remote capability boundary. The model can call tools; the user often approves signing or delegates to an agent that signs. Trust is layered: you trust the client, the client trusts the server response, and the chain only cares about the signed bytes. The friction is between moving fast (fewer checks) and reducing harm (more checks before a key is used).

Where trust breaks down

Execution tools return intent: calldata and targets, not settled state. If the only step after swap.quote is sign, you are trusting whoever built the quote (routing APIs, the MCP host, prompt injection) that the to address and data match what you think you are doing. Speed favors skipping verification. Security favors treating every transaction as untrusted until checked against independent sources and your own policy.

Syenite’s default posture

Syenite does not hold keys. It returns unsigned transactionRequest objects. That splits routing and data from custody: the dangerous step always stays on the client. The trust layer tools extend that idea: you can verify and simulate without trusting Syenite’s narrative about the tx, because tx.verify pulls Etherscan and Sourcify, and tx.simulate uses standard eth_call on public RPCs. Anyone can reproduce those checks.

Three checks, one pipeline

Identity (tx.verify). Is the callee a verified contract, a known protocol in Syenite’s registry, a proxy, or an EOA? What function selector is encoded? This answers “who am I really calling?”

Behavior (tx.simulate). Does the call revert at the current head? What gas is estimated? Is native value leaving the sender? This answers “does this tx plausibly execute as intended on today’s state?” Full token balance diffs need trace tooling; simulation still catches many reverts and gross mismatches before a signature.

Policy (tx.guard). You define max native value, allowlists, blocklists, gas caps, and whether the target must be in the curated registry. This answers “does this tx fit my agent’s mandate?” The rules are yours; the server evaluates them mechanically.

Speed vs security in practice

Each step adds latency (HTTP to explorers, RPC for call and estimateGas) and can fail on rate limits or flaky RPCs. The fast path is quote then sign. The safer path is verify, simulate, guard, then sign. Autonomous agents can default to the pipeline for high-value or unfamiliar contracts, and relax it for repeated, allowlisted routes once you have operational confidence.

Demo-style flow

1. Call swap.quote and take transactionRequest (to, data, value, from).

2. tx.verify with that to, chain, and data. Read riskFlags and protocol.syeniteAllowlisted.

3. tx.simulate with the same fields plus from. Confirm success and review gasEstimateUSD and any revertReason.

4. tx.guard with your rules (e.g. maxValueNative, requireAllowlisted: true for a conservative agent).

5. Sign only if the outcomes are acceptable. This is how Syenite encodes “trust but verify” without taking custody.
